# Задание: Бенчмарк моделей сетевого I/O в Linux/Java

## Цель

Провести сравнительное тестирование 4 моделей сетевого I/O в Linux на JDK 21.

## Модели I/O

1. **Blocking I/O** — классическая блокирующая модель, поток на соединение
2. **Non-blocking I/O** — неблокирующий I/O (Java NIO Selector)
3. **I/O Multiplexing** — epoll (обязательно), select (опционально — показать ограничения)
4. **Asynchronous I/O (io_uring)** — как аналог POSIX aio

## Параметры нагрузки

- **Соединения**: 1, 10, 100, 1 000, 10 000
- **Размер данных**: 64 B, 512 B, 4 KB, 16 KB, 64 KB, 128 KB, 512 KB, 1 MB

## Условия тестирования

- Тестирование на **localhost** (loopback) — исключить влияние сети
- **Короткоживущие соединения** (Connection: close) — каждый запрос = новое TCP-соединение
- **HTTP протокол**
- **Многопоточность** — моделируем условия, приближенные к реальной работе сервера
- **CPU pinning** через taskset (без HyperThreading):
  - 1 физическое ядро: сервер + клиент на одном ядре
  - 4 физических ядра: сервер на 2, клиент на 2
  - 8 физических ядер: сервер на 4, клиент на 4

## Метрики

1. **Пропускная способность** — MB/s, Mpps (megapackets per second)
2. **Латентность** — p50, p90, p99, p999
3. **CPU usage** — на сервере и клиенте
4. **Контекстные переключения** — voluntary / involuntary
5. **Системные вызовы** — в разбивке по типам (**наиболее важный пункт**, ожидается что io_uring значительно лучше)
6. **Задержки в ядре** — perf sched или perf record (базовые метрики, углубимся при необходимости)
7. **Потребление памяти** — RSS, буферы, аллокации (среднее)
8. **Файловые дескрипторы** — собирать во время теста

## Требования к визуализации

- Хотелось бы видеть как меняется значение каждой метрики **во время выполнения** теста
- На одном графике
- CSV на каждую конфигурацию — отдельный файл

## Реализация io_uring

- Netty с нативным транспортом io_uring (JNI) — основной вариант
- **FFM (Foreign Function & Memory API)** — свой демо-код, показать преимущества и недостатки
- Использовать **расширенные возможности**: фиксированные буферы, multishot accept

## Реализация серверов

- Тестировать через готовые решения: **Netty**
- JDK 21
- Blocking: пул потоков с ограничением
- Non-blocking: NIO Selector (фиксированный EventLoopGroup)
- Epoll: Netty native epoll transport
- io_uring: Netty native io_uring transport

## Параметры тестов

- **Warmup**: 5 секунд
- **Активное измерение**: 30 секунд
- **Повторы**: 2 на каждую конфигурацию
- **Select**: опционально, но показать ограничения (FD_SETSIZE = 1024)

## Формат отчёта

1. **Методология**: описание тестовой среды, инструментов и параметров нагрузки, сценарий воспроизведения (последовательность команд)
2. **Результаты**: CSV-файлы + визуализация в ipynb с подробными комментариями
3. **Выводы**: по результатам испытаний

## Итого: 3 deliverables

1. Предложить методику проведения испытаний
2. Провести испытания
3. Оформить выводы по результатам испытаний
